# Configuration Directory

Consolidated configuration structure for the Claude Agent Framework.

## Directory Structure

```
config/
├── repo/              # Repository-level settings and configuration
│   ├── repo-profile.yaml              # Repository identity and metadata
│   ├── repo-settings.yaml             # Repository behavior settings
│   ├── security-policy.yaml           # Security policies
│   ├── agent-bundles.json             # Agent bundle definitions
│   ├── public-release-allowlist.yaml  # Public release configuration
│   ├── error-solutions.yml            # Error resolution mappings
│   └── doc-sync-map.json              # Documentation sync mappings
│
├── agents/            # Agent configuration files
│   ├── architect.md
│   ├── backend-developer.md
│   ├── frontend-developer.md
│   ├── pm-orchestrator.md
│   ├── test-qa.md
│   └── decision-framework/
│       ├── need-agent.md
│       ├── scope-agent.md
│       ├── timing-agent.md
│       └── vision-agent.md
│
├── commands/          # Command/skill configuration files
│   ├── capture.md
│   ├── issue-triage.md
│   ├── milestone-audit.md
│   ├── pr-to-main.md
│   ├── pr-to-qa.md
│   ├── sprint-status.md
│   └── ... (additional command configs)
│
├── environment/       # Environment and runtime configuration
│   ├── environment-tiers.yaml         # Environment tier definitions
│   ├── container-capacity.env.example # Container capacity settings
│   ├── container-queue.env.example    # Container queue configuration

│   └── proxmox-n8n.yml                # Proxmox/n8n integration config
│
└── policies/          # Policy and permission files
    ├── agent-permissions.yaml         # Agent authorization policies
    ├── claude-md-policy.yaml          # CLAUDE.md validation policies
    ├── corporate-mode.yaml            # Corporate mode settings
    └── container-permission-policy.yaml # Container permission policies
```

## Subdirectory Ownership

### `config/repo/`
**Owner**: Framework core maintainers
**Purpose**: Repository-level configuration that defines the framework's behavior and identity.

Files here control:
- Repository metadata and profile
- Core security policies
- Error handling mappings
- Documentation sync configuration
- Agent bundle definitions

### `config/agents/`
**Owner**: Agent developers
**Purpose**: Configuration files for specialized agents that can be synced to deployed instances.

These files define agent behavior, capabilities, and system prompts. They are synchronized to target environments via the framework's sync mechanism.

### `config/commands/`
**Owner**: Skill/command developers
**Purpose**: Configuration and documentation for slash commands and skills.

Command configs provide:
- Skill documentation
- Usage examples
- Configuration templates
- Override instructions

### `config/environment/`
**Owner**: DevOps/infrastructure team
**Purpose**: Environment-specific configuration for runtime behavior.

Includes:
- Container orchestration settings
- Environment tier definitions
- LLM provider configuration
- Infrastructure integration configs

### `config/policies/`
**Owner**: Security/governance team
**Purpose**: Security policies, permissions, and governance rules.

Controls:
- Agent authorization boundaries
- CLAUDE.md validation rules
- Corporate mode enforcement
- Container permission policies

## Configuration Loading

Configuration files are loaded by different framework components:

- **Python modules** (`src/`): Load via `config/repo/security-policy.yaml`, `config/environment/llm-providers.yaml`
- **Scripts** (`scripts/`): Reference `config/repo/doc-sync-map.json` and other repo configs
- **Sync system** (`scripts/generate-manifest.sh`): Scans `config/` for syncable assets
- **Agents/commands**: Loaded from `config/agents/` and `config/commands/` during framework sync

## Sync Manifest

The `.sync-manifest.json` file in the repository root tracks all configuration files and their versions. This manifest is automatically regenerated by the pre-commit hook whenever files in the `config/` directory are modified.

## File Metadata

Configuration files can include metadata in frontmatter or comments:

```markdown
---
version: 1.0.0
sync_mode: overwrite
---
```

Or as comments:

```markdown
# version: 1.0.0
# sync_mode: overwrite
```

### Sync Modes

- `overwrite`: Replace the target file completely (default)
- `merge`: Merge changes with existing file
- `skip`: Skip synchronization for this file

## Migration Notes

This structure consolidates the previous `config/` and `configs/` directories:

**Previous structure**:
- `config/` - 13 repo-level config files (flat)
- `configs/` - 31 agent/command/environment configs (flat with subdirs)

**New structure**:
- Single `config/` directory with clear subdirectory ownership
- Logical grouping by purpose (repo, agents, commands, environment, policies)
- Clear ownership boundaries for each subdirectory

## Scripts

- `scripts/generate-manifest.sh`: Generates the `.sync-manifest.json` file
- Pre-commit hook: Automatically regenerates manifest when config files change
